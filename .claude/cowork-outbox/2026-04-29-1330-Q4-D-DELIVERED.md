## Q4-D-1: ApiNotActivatedMapError debug

**Hipótesis ganadora**: la D (key allow-list mismatch en runtime).
Hipótesis B (Maps Embed API) descartada — `git grep` confirma que
**no** usamos `<iframe src=".../maps/embed?...">` en ninguna parte.
Hipótesis A (quota) descartable — el error es 100% reproducible, no
intermitente.

**Root cause**: Step 0 de `/patient/request` renderiza dos
consumidores de Google Maps en paralelo:
- `<PlacesAutocomplete>` lee `NEXT_PUBLIC_GOOGLE_PLACES_KEY` y
  carga `<script src=".../api/js?key=K1&libraries=places">`.
- `<AddressMap>` (vía `@vis.gl/react-google-maps` `<APIProvider>`)
  lee `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` y carga su propio loader
  con `key=K2`.

`PLACES_KEY` (verificado por audit) tiene Maps JS + Places +
Geocoding habilitadas. `MAPS_API_KEY` tenía allow-list más estrecha.
Resultado: Google rechazaba el segundo script con
`ApiNotActivatedMapError` porque K2 no tenía Maps JS habilitado.

**Fix estructural**: una sola fuente de verdad para la public Maps key.

```ts
// lib/maps/api-key.ts (nuevo)
export function getGoogleMapsKey(): string | undefined {
  const places = process.env.NEXT_PUBLIC_GOOGLE_PLACES_KEY
  const maps = process.env.NEXT_PUBLIC_GOOGLE_MAPS_API_KEY
  return places || maps || undefined
}
```

4 callers migrados a `getGoogleMapsKey()`:
- `components/shared/address-map.tsx` (+ `libraries={['places']}` en
  `<APIProvider>` para que el sibling autocomplete encuentre
  `window.google.maps.places` ya cargado)
- `components/booking/PlacesAutocomplete.tsx`
- `app/[locale]/patient/tracking/[id]/page.tsx`
- `app/[locale]/patient/request/page.tsx` (reverse-geocode fetch)

`next.config.js`: añadido `NEXT_PUBLIC_GOOGLE_PLACES_KEY` al `env`
block para superficie pública explícita.

## Q4-D-2: reviewCount real desde DB

**DB verificada con MCP**:
```sql
SELECT
  COUNT(*) FILTER (WHERE is_public = true) AS public_count,
  ROUND(AVG(rating) FILTER (WHERE is_public = true)::numeric, 2) AS public_avg
FROM consultation_reviews;
-- public_count: 5, public_avg: 4.60
```

Coincide con el dato del inbox.

**Fix**: `MedicalOrganizationJsonLd` ahora es async server component.

```ts
async function fetchAggregateRating() {
  const supabase = createServiceRoleClient()
  const { data, error } = await supabase
    .from('consultation_reviews')
    .select('rating')
    .eq('is_public', true)
  if (error || !data || data.length < 5) return null
  const avg = data.reduce((s, r) => s + r.rating, 0) / data.length
  return { ratingValue: avg.toFixed(1), reviewCount: String(data.length) }
}
```

Cuando count < 5, `aggregateRating` se omite del schema (Google
requiere ≥5 reviews para SERP stars). Cuando count ≥ 5, se incluye
con `ratingValue` formateado a 1 decimal.

## Decisions flagged

1. **Service-role client per render**: `MedicalOrganizationJsonLd`
   crea cliente fresh cada layout render. Vercel ISR amortiza esto;
   si pasamos a streaming SSR con tráfico alto, envolver en
   `unstable_cache({ revalidate: 60 })`. No necesario en alpha.
2. **Threshold 5 reviews**: hardcoded en `fetchAggregateRating`. Si
   Google cambia la policy, una sola línea.
3. **Ambas env vars siguen expuestas** en `next.config.js`:
   backward-compat. Drop `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` post-launch
   cuando confirmemos que nadie lo lee directo.
4. **`mapId="address-picker"` / `"tracking-map"`**: placeholders.
   Google fallback a estilo default cuando el ID no existe en Cloud
   Console. Wire real Map IDs post-launch para custom styling.
