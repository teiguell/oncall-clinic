## Z-3 root cause detail

Live console mostraba `TypeError: Cannot read properties of undefined
(reading 'YJ')`. Dos issues stacked:

1. `script.onload` fires antes de que `places.Autocomplete` esté
   poblado en `window.google.maps.places`. Google's bootstrap loader
   mounts the namespace on a microtask, no en el load event.
2. El `useEffect` que construye el Autocomplete corría en mount con
   `window.google` undefined, retornaba, y nunca volvía a correr (no
   dep change tras script load).

Tres capas de fix:
- `loading=async` en script URL (param Google recommends para new loader)
- Inside `script.onload`, poll up to ~500ms for `Autocomplete` being
  a function antes de resolver el loader promise
- `scriptReady` state set después de `loadPlacesScript()` resolves;
  añadido al deps del init effect. Constructor wrapped en try/catch.

## Decisions flagged

1. **PullToRefresh threshold 70px** — paridad con iOS native (~70-80).
   Tweakable via `TRIGGER_DISTANCE` const si QA lo encuentra
   sensitive/sluggish.
2. **Browser Notification permission requested on tracking-page mount**
   (no CTA explícita). Post-payment context justifies the prompt; si
   prefieres pattern explícito, swap a `<PushSubscriber />`-style
   (Round 17-F).
3. **Notification body en ES-only** (uses `getStatusLabel()` from
   lib/utils/index.ts). Not a regression but EN-locale users ven
   labels en español. i18n sweep próximo.
4. **PullToRefresh listeners en `window`** vs container ref. Simpler;
   trade-off: lista interna scrollable (e.g. doctor pendingRequests)
   no triggerea pull mientras esté en su scrollTop interno. Acceptable.
5. **Refactor `refetch` en ClinicDashboardLive** a useCallback so
   PullToRefresh + realtime + 30s poll share the same fetch path.
6. **Z-9 eyebrow** swapped from "Confían en nosotros" → "Próximas
   clínicas asociadas" — sets honest expectations en lugar de implicar
   partners ya establecidos.
