## Bug 14F-3 (P0) — service-role bypass for RLS

**Pattern applied uniformly to 5 endpoints:**

```ts
const cookieSupabase = await createClient()
const { data: { user } } = await cookieSupabase.auth.getUser()
const bypass = getBypassUser()
const supabase = !user && AUTH_BYPASS && bypass
  ? createServiceRoleClient()  // bypass → RLS-free
  : cookieSupabase              // production → RLS via auth.uid()
```

**Why it's safe in production:** the service-role escape is gated on `AUTH_BYPASS=true`, which is only set on Vercel preview/audit envs (per `lib/auth-bypass.ts` design). On real production with a logged-in user, `user` is non-null → `cookieSupabase` is used → RLS still enforces row ownership.

**Endpoints patched:**

| Endpoint | RLS-protected ops | Now works in bypass |
|---|---|---|
| `/api/stripe/checkout` | INSERT consultations (TEST + production paths) | ✓ |
| `/api/consultations/[id]/accept` | UPDATE consultations + INSERT notifications_log | ✓ |
| `/api/consultations/[id]/complete` | UPDATE consultations + INSERT pending_payouts | ✓ |
| `/api/clinic/register` | INSERT clinics + UPDATE profiles | ✓ |
| `/api/doctor/stripe-connect/init` | UPDATE doctor_profiles.stripe_account_id | ✓ |

**New helper:** `lib/supabase/service.ts` exports `createServiceRoleClient()`. Throws if `SUPABASE_SERVICE_ROLE_KEY` env var is missing — fail-loud rather than silently fall back to a broken client.
